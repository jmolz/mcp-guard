# Feature: Phase 1 — Foundation

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Build the foundational infrastructure for MCP-Guard: project scaffolding, config system, daemon lifecycle, thin bridge process, and basic MCP proxy that transparently passes all messages between client and upstream server. This phase delivers a working end-to-end proxy with no security interceptors yet — just a transparent pipe through the daemon.

## User Story

As a developer using MCP servers with AI coding tools, I want to point my MCP client at MCP-Guard instead of a raw server, so that all traffic flows through a central daemon that I can later add security policies to.

## Success Criteria

- [ ] `pnpm lint && pnpm typecheck && pnpm test && pnpm build` all pass with zero errors
- [ ] `mcp-guard start` launches a daemon that listens on a Unix socket
- [ ] `mcp-guard stop` gracefully shuts down the daemon
- [ ] `mcp-guard connect --server <name>` spawns a thin bridge that proxies stdio to daemon
- [ ] Daemon auto-starts on first `mcp-guard connect` if not already running
- [ ] Bridge authenticates to daemon via daemon.key
- [ ] All MCP message types pass through transparently (tools/list, tools/call, resources/read, etc.)
- [ ] Graceful shutdown completes in-flight requests before exiting
- [ ] `mcp-guard validate` checks config file without starting daemon
- [ ] `mcp-guard status` shows daemon health when running
- [ ] `mcp-guard health` returns exit code 0 when daemon is healthy, 1 otherwise
- [ ] No regressions — all tests pass throughout

### Task 10: OS Identity (Peer Credentials via koffi)

**File:** `src/identity/os-identity.ts`
**Action:** CREATE
**Description:** Verify the OS identity of connecting bridge processes using Unix socket peer credentials.

```typescript
export interface PeerCredentials {
  uid: number;
  gid: number;
  pid?: number; // Available on Linux, macOS via LOCAL_PEERPID
}

export function getPeerCredentials(socketFd: number): PeerCredentials;
export function verifyPeerIsCurrentUser(creds: PeerCredentials): boolean;
```

**macOS implementation:**
- Use koffi to call `getpeereid(int, _Out_ int*, _Out_ int*)` from `libc.dylib` for UID/GID
- Use koffi `getsockopt` with `LOCAL_PEERPID` (level=0, optname=2) for PID

**Linux implementation:**
- Use koffi to call `getsockopt` with `SO_PEERCRED` (SOL_SOCKET=1, SO_PEERCRED=17) and `struct ucred { pid: int, uid: int, gid: int }`

**Platform detection:** `process.platform === 'darwin'` vs `'linux'`

`verifyPeerIsCurrentUser` compares `creds.uid` against `process.getuid()`.

### Task 14: Unix Socket Server

**File:** `src/daemon/socket-server.ts`
**Action:** CREATE
**Description:** The daemon's Unix socket server that bridges connect to. Handles connection authentication, message framing, and peer credential verification.

```typescript
export interface SocketServerOptions {
  socketPath: string;
  daemonKey: Buffer;
  onConnection: (conn: AuthenticatedConnection) => void;
}

export interface AuthenticatedConnection {
  id: string;
  uid: number;
  pid?: number;
  send(message: DaemonMessage): void;
  onMessage(handler: (message: BridgeMessage) => void): void;
  close(): void;
}

export function createSocketServer(options: SocketServerOptions): SocketServer;
```

**Protocol:**
1. Bridge connects to Unix socket
2. Bridge sends auth message: `{ type: 'auth', key: <hex-encoded daemon.key> }`
3. Daemon verifies key (constant-time) + peer UID (via koffi)
4. Daemon responds: `{ type: 'auth_ok' }` or closes connection
5. Length-prefixed JSON messages after auth: `<4-byte big-endian length><JSON payload>`

**Security:**
- Socket file created with 0600 permissions
- Old socket file removed on startup (stale from crash)
- Peer credentials verified on every connection
- Unauthenticated connections closed after 5s timeout

**Validate:** `pnpm typecheck`

### Task 22: Bridge Process

**File:** `src/bridge/index.ts`
**Action:** CREATE
**Description:** The thin bridge process. This MUST stay small — zero policy logic, zero upstream connection code. ~50 lines of actual logic.

```typescript
export async function startBridge(serverName: string, configPath?: string): Promise<void>;
```

**What the bridge does (and nothing else):**
1. Check if daemon is running; if not, auto-start it
2. Connect to daemon Unix socket
3. Send auth message with daemon.key (hex-encoded)
4. Wait for `auth_ok` response (or exit on `auth_fail`)
5. Pipe stdin → wrap as `{ type: 'mcp', server, data }` → send to daemon
6. Receive daemon messages → if `{ type: 'mcp' }` → write `data` to stdout
7. On daemon `shutdown` message → exit 1
8. On stdin close → close socket → exit 0
9. On socket close → exit 1

**Message framing:** Length-prefixed JSON (4-byte big-endian uint32 length, then JSON payload), matching the socket server protocol.

**Structural guarantee:** This file must not import anything from `src/interceptors/`, `src/proxy/`, `src/pii/`, `src/audit/`, `src/config/schema.ts`, or `src/storage/`. It only imports from `src/bridge/`, `src/identity/daemon-key.ts`, `src/daemon/auto-start.ts`, `src/constants.ts`, and `src/errors.ts`.

**Depends on:** Task 15, Task 8, Task 21
**Validate:** `pnpm typecheck`

**Phase 1 commands:**

- `mcp-guard start` — Start daemon in foreground (or `--daemon` / `-d` for background)
  - Loads config, calls `startDaemon(config)`
  - `--config <path>` flag for custom config path

- `mcp-guard stop` — Stop running daemon
  - Reads PID from PID file, sends SIGTERM
  - Waits for process to exit (poll, max 10s)
  - If still running after timeout, SIGKILL

- `mcp-guard connect --server <name>` — Start bridge
  - Calls `startBridge(serverName, configPath)`
  - `--config <path>` flag

- `mcp-guard status` — Show daemon status
  - Checks if daemon is running (socket probe)
  - If running: prints uptime, connected bridges, server statuses
  - If not running: prints "Daemon is not running"

- `mcp-guard health` — Liveness check
  - Exit code 0 if daemon is healthy, 1 if not
  - No output on success (designed for scripts/monitoring)

- `mcp-guard validate` — Validate config
  - Loads and validates config (same as daemon startup)
  - Prints validation result
  - Exit code 0 if valid, 1 if invalid

## Notes

- **SSE transport is deferred to Phase 4** — Phase 1 only handles stdio upstream servers
- **No interceptor pipeline yet** — all messages pass through transparently. Phase 2 adds the pipeline.
- **No audit logging yet** — the audit tap is structural and will be added in Phase 2 alongside the pipeline
- **Dashboard is deferred to Phase 4** — the health endpoint (`/healthz`) could be added here but is not required for Phase 1 validation
- **koffi peer credentials** — the exact API for output parameters may need adjustment during implementation. The koffi docs show `_Out_` qualifiers but the actual buffer/pointer handling might differ. Test on the target platform.
- **MCP SDK imports** — the SDK uses deep imports like `@modelcontextprotocol/sdk/server/index.js` and `@modelcontextprotocol/sdk/client/index.js`. Check the actual package exports map after installation.
- **Length-prefixed framing** — chosen over newline-delimited JSON because MCP messages can contain newlines in string values. 4-byte big-endian uint32 prefix gives a 4GB max message size which is more than sufficient.
