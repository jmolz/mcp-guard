# Feature: Phase 5B — Launch

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Phase 5B transforms MCP-Guard from a working development project into a publishable, installable product. It adds the `mcp-guard init` CLI command, Docker packaging, npm publish configuration, CI/CD release pipelines, benchmark chart/table generation, Tier 2 compatibility tests against real open-source MCP servers, comprehensive documentation (README, CONTRIBUTING, SECURITY, CODE_OF_CONDUCT, CHANGELOG), and a blog post outline.

## User Story

As an MCP user, I want to `npm install -g mcp-guard` or `docker run ghcr.io/jmolz/mcp-guard`, run `mcp-guard init` to generate a config from my existing MCP client setup, and immediately have a production-grade security proxy — with published benchmarks proving its effectiveness.

### Workstream 3: `mcp-guard init` Command

### Task 5: Implement init command

**File:** `src/cli/init.ts`
**Action:** CREATE
**Description:** The `init` command discovers existing MCP client configurations and generates a `mcp-guard.yaml` that proxies them through MCP-Guard.

**Discovery locations (in priority order):**
1. **Claude Desktop:** `~/Library/Application Support/Claude/claude_desktop_config.json` (macOS), `%APPDATA%/Claude/claude_desktop_config.json` (Windows)
2. **Claude Code:** `~/.claude.json` or `~/.config/claude/settings.json`
3. **Cursor:** `~/.cursor/mcp.json` or `~/.config/cursor/mcp.json`
4. **VS Code (Copilot):** `~/.vscode/mcp.json` or workspace `.vscode/mcp.json`
5. **Windsurf:** `~/.codeium/windsurf/mcp_config.json`

**For each discovered config:**
- Parse JSON, extract `mcpServers` entries
- For each server: extract `command`, `args`, `env`
- Deduplicate servers with same command+args across clients
- Generate `mcp-guard.yaml` with all discovered servers

**Output:** A valid `mcp-guard.yaml` with sensible defaults:
- `auth.mode: os` (default, zero-config)
- `daemon.home: ~/.config/mcp-guard`
- Each server with its transport, command, args
- Comments explaining each section
- Print instructions on how to update each client config to use MCP-Guard

**Options:**
- `--output <path>` (default: `mcp-guard.yaml`)
- `--dry-run` (print to stdout, don't write)
- `--client <name>` (only scan a specific client: `claude-desktop`, `claude-code`, `cursor`, `vscode`, `windsurf`)

**IMPORTANT:** Never modify client config files. Only generate the MCP-Guard config and print instructions.
**Depends on:** none
**Validate:** `pnpm typecheck`

### Workstream 5: Tier 2 Compatibility Tests

### Task 13: Create Tier 2 compatibility test infrastructure

**File:** `tests/compat/tier2.test.ts`
**Action:** CREATE
**Description:** Test MCP-Guard against real open-source MCP servers. These tests spawn actual MCP server processes, configure MCP-Guard to proxy them, and verify end-to-end message flow.

**Servers to test (available without external credentials):**
1. `@modelcontextprotocol/server-filesystem` — file operations on a temp directory
2. `@modelcontextprotocol/server-memory` — in-memory knowledge graph
3. `@modelcontextprotocol/server-sqlite` — SQLite operations on a temp database (note: **this is a different SQLite instance from MCP-Guard's own DB**)
4. `@modelcontextprotocol/server-everything` — meta-server exposing all MCP capabilities
5. `@modelcontextprotocol/server-sequential-thinking` — thinking/reasoning tool

**For servers requiring credentials (test when env vars are present, skip otherwise):**
6. `@modelcontextprotocol/server-github` — `GITHUB_TOKEN`
7. `@modelcontextprotocol/server-brave-search` — `BRAVE_API_KEY`
8. `@modelcontextprotocol/server-fetch` — no credentials, but needs network
9. `@modelcontextprotocol/server-git` — local git operations

**Test pattern for each server:**
```typescript
describe('filesystem server', () => {
  it('proxies initialize + tools/list', async () => {
    // 1. Start daemon with server configured
    // 2. Connect via bridge socket
    // 3. Send initialize → verify response
    // 4. Send tools/list → verify tools returned
    // 5. Optionally: send one tools/call → verify result
    // 6. Check audit log has entries
  });
});
```

**Infrastructure:**
- Generate a temp config YAML per test with the server configured
- Use `startDaemon(config)` in-process (same pattern as benchmark/E2E tests)
- Connect via Unix socket and frame protocol (reuse `tests/fixtures/framing.ts`)
- Use `describe.skipIf()` for servers needing credentials
- Clean up temp dirs in `afterAll`
- Set reasonable timeout (30s per server — they need to npm install on first run)

**Key: these tests prove MCP-Guard proxies real servers transparently.** The benchmark mock servers proved the interceptor pipeline; Tier 2 proves the proxy doesn't break real server protocols.
**Depends on:** none (uses existing daemon infrastructure)
**Validate:** `pnpm vitest run tests/compat/tier2.test.ts`

## Notes

### Design Decisions

- **TypeScript SVG generation over Python matplotlib**: Avoids adding Python as a runtime/dev dependency. SVG template strings are simple for bar charts and adequate for our needs. If charts need to become more sophisticated later, we can switch to a Node.js charting library (e.g., vega-lite).
- **`npx -y` for Tier 2 servers**: Prefer `npx -y @modelcontextprotocol/server-X` over devDependencies to avoid bloating the lockfile and to test the same install path users experience.
- **Tier 2 scope: 5-9 servers, not 20**: The PRD targets "20/20 server compatibility" but many popular servers require external credentials (GitHub token, API keys, cloud accounts). We test all credential-free servers and conditionally test credential-requiring ones when env vars are set. Full 20/20 validation happens in CI with secrets configured.
- **Blog post is outline only**: The full blog post is a writing task, not a code task. The outline provides structure and references to embed benchmark data.

### Risks

1. **Native module Docker build**: `koffi` and `better-sqlite3-multiple-ciphers` compile native binaries. `node:22-slim` should work but if not, may need `node:22` (larger image) or explicit build dependencies.
2. **Tier 2 test flakiness**: Real MCP servers may have startup timing issues. Use retries and generous timeouts.
3. **npm publish permissions**: First publish requires `npm login` and org setup. The workflow is correct but the first publish may need manual intervention.
