# Feature: Phase 2 — Interceptor Pipeline + Auth

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Build the security core of MCP-Guard: a fixed-order interceptor pipeline that inspects every MCP message, with authentication, rate limiting, permission scoping, and a structural audit tap that logs all interactions. This phase transforms MCP-Guard from a transparent proxy into an actual security tool.

## User Story

As a developer using MCP servers with AI coding tools, I want every MCP request to pass through a security pipeline that authenticates identity, enforces rate limits, checks permissions, and logs all interactions, so that I have visibility and control over what my AI tools do with MCP servers.

## Success Criteria

- [ ] `pnpm lint && pnpm typecheck && pnpm test && pnpm build` all pass with zero errors
- [ ] Every MCP message flows through the interceptor pipeline before reaching upstream
- [ ] Auth interceptor resolves OS identity (uid → username) and maps to roles from config
- [ ] Rate limit interceptor enforces token bucket limits per server/tool, persisted to SQLite
- [ ] Permission interceptor blocks denied tools/resources and allows only permitted ones
- [ ] Capability filtering removes denied tools/resources from `tools/list` and `resources/list` responses
- [ ] Audit tap logs all messages (including blocked ones) to SQLite + structured stdout
- [ ] `mcp-guard logs` CLI queries audit log with filtering
- [ ] Interceptor throw → request BLOCKED (fail-closed)
- [ ] Interceptor timeout → request BLOCKED (fail-closed)
- [ ] No regressions — all 43 existing tests still pass

### Task 4: Rate Limit Store

**File:** `src/storage/rate-limit-store.ts`
**Action:** CREATE
**Description:** Token bucket rate limiter backed by SQLite. Tokens refill continuously. State persists across daemon restarts.

```typescript
export interface RateLimitStore {
  /** Try to consume one token. Returns true if allowed, false if rate limited. */
  tryConsume(key: string, config: { maxTokens: number; refillRate: number }): boolean;
  /** Get remaining tokens for a key */
  getRemaining(key: string): number | null;
  /** Clean up expired entries */
  cleanup(olderThan: Date): void;
}

export function createRateLimitStore(db: Database): RateLimitStore;
```

Token bucket algorithm:
1. Read current token count and last refill time
2. Calculate elapsed time, add `elapsed * refillRate` tokens (capped at maxTokens)
3. If tokens >= 1, consume one and update. Return true.
4. If tokens < 1, return false (rate limited).
5. All in a single SQLite transaction for atomicity.

### Task 6: Pipeline Runner

**File:** `src/interceptors/pipeline.ts`
**Action:** CREATE
**Description:** Runs the interceptor chain with timeout enforcement and fail-closed semantics. This is the security core — every design decision defaults to BLOCK.

```typescript
export interface PipelineOptions {
  interceptors: Interceptor[];
  timeout: number; // ms, per interceptor
  logger: Logger;
}

export function createPipeline(options: PipelineOptions): {
  execute(ctx: InterceptorContext): Promise<PipelineResult>;
};
```

**Execution logic:**
1. For each interceptor in order:
   a. Wrap `interceptor.execute(ctx)` in `Promise.race` with timeout
   b. If timeout expires → `{ action: 'BLOCK', reason: 'Interceptor {name} timed out' }`
   c. If interceptor throws → `{ action: 'BLOCK', reason: 'Interceptor {name} failed: {error}' }`
   d. If decision is `BLOCK` → short-circuit, skip remaining interceptors
   e. If decision is `MODIFY` → apply modified params to context for next interceptor
   f. If decision is `PASS` → continue
2. Record each decision with timing info
3. Return `PipelineResult` with all decisions, final params, and allowed flag

**Key invariants:**
- A thrown error MUST result in BLOCK — never silently pass
- Timeout MUST result in BLOCK — never silently pass
- MODIFY decisions: validate that only `params`/`content` changed (reject mutations to `method` or `toolName`)
- Pipeline result always returned — even on error

**Depends on:** Task 1
**Validate:** `pnpm typecheck`

### Task 20: Wire Pipeline into Daemon

**File:** `src/daemon/index.ts`
**Action:** MODIFY
**Description:** Replace direct proxy passthrough with pipeline → proxy → audit tap flow.

**Current flow (line 62-69):**
```typescript
onConnection: (conn) => {
  conn.onMessage(async (msg) => {
    if (msg.type === 'mcp') {
      const response = await proxyServer.handleMessage(msg.data, msg.server);
      conn.send({ type: 'mcp', data: response });
    }
  });
}
```

**New flow:**
```typescript
onConnection: (conn) => {
  const identity = resolveIdentity(conn.uid, conn.pid, config);

  conn.onMessage(async (msg) => {
    if (msg.type === 'mcp') {
      const ctx: InterceptorContext = {
        message: { method: msg.data.method, params: msg.data.params },
        server: msg.server,
        identity,
        direction: 'request',
        metadata: { bridgeId: conn.id, timestamp: Date.now() },
      };

      // Run pipeline
      const startTime = Date.now();
      const result = await pipeline.execute(ctx);

      // Audit tap records everything (including blocks)
      auditTap.record({
        bridgeId: conn.id,
        server: msg.server,
        method: msg.data.method ?? 'unknown',
        direction: 'request',
        identity,
        toolOrResource: extractToolOrResource(msg.data),
        pipelineResult: result,
        latencyMs: Date.now() - startTime,
      });

      if (!result.allowed) {
        // Return JSON-RPC error to client
        conn.send({ type: 'mcp', data: {
          jsonrpc: '2.0',
          id: msg.data.id,
          error: { code: -32600, message: result.decisions.find(d => d.decision.action === 'BLOCK')?.decision.reason ?? 'Blocked' },
        }});
        return;
      }

      // Forward to upstream (with potentially modified params)
      const upstreamMsg = result.finalParams
        ? { ...msg.data, params: result.finalParams }
        : msg.data;
      const response = await proxyServer.handleMessage(upstreamMsg, msg.server);

      // Apply capability filtering to list responses
      if (msg.data.method === 'tools/list') {
        response.result.tools = filterToolsList(response.result.tools, ...);
      }

      conn.send({ type: 'mcp', data: response });
    }
  });
}
```

Also create the pipeline, audit store, audit tap, and rate limit store during daemon startup (before the socket server).

## Notes

- **PII detection is deferred to Phase 3** — the pipeline has a slot for the PII interceptor but it won't be implemented yet. The pipeline runner should handle an empty PII interceptor gracefully.
- **Token/OAuth auth is deferred to Phase 4** — Phase 2 implements OS identity + API key only.
- **Config extends/merge is deferred to Phase 4** — Phase 2 uses local config only, no floor-based merging yet.
- **Sampling guard is deferred to Phase 3** — the capability filtering framework built here will support it.
- **Response-direction PII scanning is deferred to Phase 3** — Phase 2 pipeline is request-only.
- **The `proxyServer.handleMessage` return type** will need adjustment to return structured data (not just JsonRpcMessage) so the capability filter can modify list responses. Check the actual return shape before implementing Task 14/20.
- **Rate limit store cleanup** — add a periodic cleanup call during daemon tick or shutdown, not per-request.
