# Feature: Phase 3 — PII Detection + Sampling Guard

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Add content-level security to MCP-Guard: a pluggable PII detection system that scans both requests (before upstream) and responses (after upstream), with configurable per-type directional actions (block/redact/warn), plus a sampling guard that prevents MCP servers from using `sampling/createMessage` unless explicitly enabled. This is the phase where MCP-Guard transitions from access control to data protection.

## User Story

As a **security engineer**, I want **PII detected and redacted from MCP traffic** in both directions, with different policies per PII type and direction, so that sensitive data doesn't leak to third-party MCP servers AND so that upstream server responses don't expose PII to LLM clients without policy control.

As a **security engineer**, I want to **block MCP servers from using sampling/createMessage** to inject prompts into the user's LLM, so that compromised servers can't manipulate agents — unless I explicitly enable it.

## Success Criteria

- [ ] PIIDetector interface implemented with pluggable registry
- [ ] Built-in regex detector covers: emails, phones, SSNs, credit cards (Luhn-validated), AWS keys, GitHub tokens
- [ ] Directional PII actions: per-type request vs response actions (block/redact/warn)
- [ ] Custom PII types definable in config with regex patterns
- [ ] Redactor returns new string, original value immediately discarded, never stored
- [ ] PII interceptor runs bidirectionally (request + response)
- [ ] Audit logs record PII type + action, never the original value
- [ ] Sampling guard blocks `sampling/createMessage` by default
- [ ] Sampling capability removed from initialize response when disabled
- [ ] Confidence threshold configurable (default 0.8)
- [ ] Fail-closed: detector error/timeout → BLOCK
- [ ] All validation commands pass with zero errors
- [ ] No regressions in existing 103 tests

  server: string;

### Task 4: Redactor

**File:** `src/pii/redactor.ts`
**Action:** CREATE
**Description:** Implements span-based redaction of PII matches. The redactor takes a string and an array of PIIMatch, replaces matched spans with type-labeled markers, and returns a NEW string. The original is never mutated.

**Core function:**
```typescript
/**
 * Replace PII matches in a string with redaction markers.
 * Returns a NEW string — the original is not mutated.
 * Process spans in reverse order to preserve offsets.
 */
export function redactString(content: string, matches: PIIMatch[]): string
```

Replacement format: `[REDACTED:type]` (e.g., `[REDACTED:email]`, `[REDACTED:ssn]`)

**Structured object scanning and redaction:**
```typescript
/**
 * Recursively walk a JSON-compatible value, scanning every string with the
 * provided detector function, collecting matches, and optionally redacting.
 *
 * Returns { matches, redacted } where:
 * - matches: all PIIMatch found, with `path` indicating where in the structure
 * - redacted: deep clone with matched spans replaced (original untouched)
 */
export function scanAndRedact(
  value: unknown,
  detect: (content: string) => PIIMatch[],
  shouldRedact: boolean,
): ScanResult
```

**Key requirements:**
- MUST NOT mutate the input value — always return a new object
- Walk arrays and plain objects recursively
- Only scan string values (skip numbers, booleans, null)
- Track the JSON path for each string scanned (e.g., `"arguments.message"`)
- Handle overlapping matches: sort by start, skip matches that overlap with a previous one
- If `shouldRedact` is false, still return matches but `redacted` equals a deep clone of the input

**Validate:** `pnpm typecheck`

### Task 5: Detector Registry

**File:** `src/pii/registry.ts`
**Action:** CREATE
**Description:** The registry holds all registered PIIDetector instances (built-in + custom) and provides a single `scan()` method that runs all detectors and aggregates results.

```typescript
export interface PIIRegistry {
  /** Run all detectors against the content, return aggregated matches above confidence threshold */
  scan(content: string, ctx: DetectionContext): PIIMatch[];
}

export function createPIIRegistry(config: PIIConfig): PIIRegistry
```

**Creation logic:**
1. Always register the built-in regex detector
2. For each `config.custom_types`, create a custom detector from the regex patterns
3. Store the confidence threshold from config

**scan() logic:**
1. Run each detector's `detect()` method
2. Aggregate all matches
3. Filter by `config.confidence_threshold`
4. Sort by `start` position
5. Deduplicate overlapping spans (keep highest confidence)

**Custom detector creation:** For each custom type in config, compile the regex patterns at registry creation time (not per-scan). If a regex is invalid, log a warning and skip it (don't throw — this matches the permissions pattern where invalid regex returns no match).

### Task 7: PII Detection Interceptor

**File:** `src/interceptors/pii-detect.ts`
**Action:** CREATE
**Description:** The PII interceptor uses the registry to scan message content and applies directional actions (block/redact/warn) based on config. It runs on BOTH request and response directions.

```typescript
export function createPiiInterceptor(
  registry: PIIRegistry,
  config: McpGuardConfig,
): Interceptor
```

**Request direction logic (scanning `ctx.message.params`):**
1. If `!config.pii.enabled` → PASS
2. Extract content to scan from `ctx.message.params` using `scanAndRedact()`
3. If no matches → PASS
4. Determine the strictest action across all matched PII types using `config.pii.actions[match.type].request`
5. Action resolution (strictest wins): `block` > `redact` > `warn`
6. If strictest is `block` → BLOCK with reason listing detected types (NOT values)
7. If strictest is `redact` → MODIFY with redacted params from `scanAndRedact()`
8. If strictest is `warn` → PASS with metadata listing detected types

**Response direction logic (scanning the upstream response):**
- Same flow but uses `config.pii.actions[match.type].response` for action lookup
- Response content to scan comes from `ctx.message.params` (which the daemon sets to the response result)
- If strictest is `block` → BLOCK (response not delivered to client)
- If strictest is `redact` → MODIFY with redacted response content
- If strictest is `warn` → PASS with metadata

**Metadata on all decisions (for audit):**
```typescript
metadata: {
  piiDetections: matches.map(m => ({
    type: m.type,
    action: resolvedAction,
    // NO value field — original value is never stored
  })),
}
```

**Key requirements:**
- Default action for unknown PII types (from custom detectors): `redact` for request, `warn` for response
- Malformed messages (no params) → PASS (other interceptors already validate this)
- Detector errors → BLOCK (fail-closed, per security rules)
- Input cap: skip scanning if serialized content exceeds 1MB (return PASS with warning in metadata — scanning multi-MB payloads would be too slow)

**Depends on:** Task 1, Task 4, Task 5, Task 6
**Validate:** `pnpm typecheck`

### Task 8: Sampling Guard Interceptor

**File:** `src/interceptors/sampling-guard.ts`
**Action:** CREATE
**Description:** Blocks `sampling/createMessage` unless explicitly enabled for the server. This is a structural security guarantee — sampling is disabled by default (fail-closed).

```typescript
export function createSamplingGuardInterceptor(
  config: McpGuardConfig,
): Interceptor
```

**Logic:**
1. If `ctx.message.method !== 'sampling/createMessage'` → PASS (only gates sampling)
2. Look up server config: `config.servers[ctx.server]?.policy.sampling`
3. If `!sampling.enabled` → BLOCK with reason `'Sampling is disabled for server "${ctx.server}"'` and code `'SAMPLING_DISABLED'`
4. If `sampling.enabled` → PASS

Note: Sampling rate limiting (`sampling.rate_limit`) is handled by the existing rate-limit interceptor in Phase 4 (not in scope here). The sampling guard only gates on/off.

## Notes

### Design Decisions

1. **Separate response pipeline** rather than making all interceptors direction-aware. Only PII needs to run on responses — Auth, RateLimit, Permissions are request-only. A separate pipeline keeps the architecture clean and avoids adding `if (direction === 'request')` checks to every interceptor.

2. **No new migration needed.** The existing `audit_logs.interceptor_decisions` JSON column stores the PII metadata through the new `metadata` field on InterceptorDecision. The `params_summary` column remains null (Phase 3 doesn't populate it — that's a future enhancement for audit query UX).

3. **Synchronous detection.** The PIIDetector interface uses synchronous `detect()` (not async) because regex-based detection is CPU-bound and fast. If a future LLM-based detector needs async, the interface can be changed then — YAGNI.

4. **PII config at top level, not per-server.** PII type definitions and actions are global. Per-server PII overrides could be added in Phase 4 but are out of scope here. The `pii.enabled` flag gates all PII scanning.

5. **Sampling rate limiting deferred.** The sampling config includes `rate_limit` and `max_tokens` fields for future use, but Phase 3 only implements the on/off gate. Rate limiting for sampling uses the same token bucket infrastructure and can be wired in Phase 4.

### Potential Follow-Up

- Per-server PII action overrides
- Sampling rate limiting via existing token bucket
- LLM-based PII detector plugin
- Streaming/sliding-window PII detection for SSE transport
- `params_summary` population in audit entries (redacted summaries)
