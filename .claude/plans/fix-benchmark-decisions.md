## Overview

The full benchmark suite reports 55% detection rate and 57.5% false positive rate due to rate limit state contamination between categories, legitimate traffic exceeding rate limits, and several category-specific expected decision bugs. This is a benchmark accuracy bug, not a security logic bug — all 362 unit/integration tests pass.

### Patterns to Follow

**Expected decision rules (from `.claude/rules/benchmarks.md`):**
- Request-side PII `redact` action → **PASS** (transparent to client, no markers in response)
- Response-side PII `redact` action → **MODIFY** (client sees `[REDACTED:type]` markers)
- PII `block` action (ssn, credit_card on request) → **BLOCK**
- Denied tool on its server → **BLOCK**
- Denied tool on a different server → **PASS**
- `*` glob matches `[^/]*` (single-level, not recursive)
- Rate limits keyed by `(server, username, tool)` for per-tool, `(server, username)` for server RPM

## Root Cause Analysis

### Primary Bug: Rate Limit State Contamination (affects ALL categories)

The security benchmark runs ONE daemon for all categories. Execution order:
1. **Burst groups run first** — each sends 65 requests per server, consuming all 60 tokens
2. **Regular categories run next** — token bucket is depleted, refill is 1/sec
3. **PASS-expected scenarios get BLOCK'd** by rate limiter → counted as detection failures

Example: `capability_probe` has ~628 PASS expectations across 8 servers. After burst groups drain the buckets, requests arrive faster than 1/sec refill. Result: ~600 scenarios return BLOCK instead of PASS → 5.8% "detection."

**Fix:** Reset the rate limit store between burst groups and regular categories in `run-security.ts`.

### Secondary Bug: Legitimate Traffic Exceeds Rate Limits

`run-legitimate.ts` sends 10,168 requests at max speed through one socket per server. With 60 RPM limit (1 token/sec refill), only the first 60 requests per server pass. The remaining ~1,200 per server are BLOCKED → counted as false positives.

**Fix:** Either disable rate limits for legitimate benchmark config OR pace requests to stay within limits OR reset rate limit state before legitimate suite.

### Category-Specific Bugs

These are secondary — fix the rate limit contamination first, then re-run to see which categories still have issues.

1. **pii_response_leak**: All 480 scenarios expect MODIFY. If `_benchmark_pii` tool response doesn't contain `[REDACTED` markers after pipeline, `inferDecision()` returns PASS. Need to verify the mock server response actually gets redacted.

2. **pii_evasion**: `valueChanged` check may be incorrect for some PII type + technique combinations (e.g., Cyrillic vowel replacement on digit-only SSN).

3. **resource_traversal**: `matchesDeniedResource()` in the generator may not perfectly mirror the actual permissions interceptor's glob matching. Discrepancies produce wrong expected decisions.

4. **capability_probe**: Most scenarios correctly expect PASS (testing that non-attacks pass). The low "detection" rate was caused by rate limiting, not wrong expectations. After rate limit fix, this category should score ~95%+.

## Implementation Tasks

Execute in order, top to bottom. Each task is atomic.

### Task 1: Reset Rate Limit Store Between Security Phases

**File:** `benchmarks/security/run-security.ts`
**Action:** MODIFY
**Description:** After burst groups complete and before running regular categories, reset the rate limit store in the daemon's SQLite database. This prevents depleted token buckets from contaminating subsequent categories. Add a function `resetRateLimits(db)` that runs `DELETE FROM rate_limits;` on the daemon's database. Call it after the burst loop completes (after line ~224 in current code) and before the regular category loop begins.

Access the database through the daemon handle — check `DaemonHandle` for a `getDb()` method or similar. If no direct DB access is available, add a `resetRateLimits()` method to `DaemonHandle` in `src/daemon/index.ts` (but prefer keeping changes in benchmarks/ if possible).

**Depends on:** none
**Validate:** `pnpm benchmark:security -- --min-detection 0.85` — detection rate should improve significantly from 55%

### Task 2: Fix Legitimate Traffic Rate Limit Issue

**File:** `benchmarks/legitimate/run-legitimate.ts`
**Action:** MODIFY
**Description:** The legitimate traffic benchmark should measure false positives from PII detection, permissions, and auth — NOT from rate limiting. The fix: reset the rate limit store at the start of the legitimate benchmark run, before sending any legitimate requests. Use the same `resetRateLimits()` approach from Task 1.

Alternatively, if rate limit state is isolated (separate daemon instance), the issue is that legitimate traffic itself exceeds rate limits by sending at max speed. In that case, either:
- (a) Use a benchmark config without rate limits for the legitimate suite (create `benchmarks/configs/legitimate-benchmark.yaml` with high RPM values)
- (b) OR pace requests with a small delay to stay within limits
- (c) OR disable rate limits in the config used for legitimate testing

Option (a) is cleanest — create a separate config with rate limits disabled or set very high (100000 RPM). The legitimate runner already accepts a config path parameter.

**Depends on:** none (independent of Task 1)
**Validate:** `pnpm benchmark:legitimate -- --min-detection 0.85` — FP rate should drop from 57.5% to <0.1%
