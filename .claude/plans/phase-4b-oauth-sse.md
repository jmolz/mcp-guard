# Feature: Phase 4B — OAuth 2.1 Token Auth + SSE/Streamable HTTP Transport

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Add OAuth 2.1 token-based authentication and HTTP transport support (SSE + Streamable HTTP) to MCP-Guard. This enables enterprise identity providers to authenticate MCP users via JWT tokens with claims-to-role mapping, and allows MCP-Guard to proxy HTTP-based upstream MCP servers — not just stdio.

## User Story

As an enterprise security admin, I want to authenticate MCP users via my organization's OAuth 2.1 identity provider and connect to HTTP-based MCP servers, so that I can enforce role-based policies using existing identity infrastructure and proxy both local and remote MCP servers.

## Success Criteria

- [ ] `auth.mode: 'oauth'` validates JWT tokens via JWKS and maps claims to roles
- [ ] `mcp-guard auth login` initiates OAuth 2.1 Authorization Code + PKCE flow
- [ ] `mcp-guard auth status` shows current token identity and expiry
- [ ] `mcp-guard auth logout` removes stored tokens
- [ ] OAuth tokens passed from bridge to daemon via `_bearer_token` in message params
- [ ] Auth interceptor strips `_bearer_token` before forwarding upstream (like `_api_key`)
- [ ] Invalid/expired/missing tokens → BLOCK (fail-closed)
- [ ] `transport: 'sse'` connects to upstream via `SSEClientTransport`
- [ ] `transport: 'streamable-http'` connects via `StreamableHTTPClientTransport`
- [ ] HTTP transport passes auth headers (Bearer token or API key) to upstream
- [ ] SSE/HTTP reconnection with configurable backoff
- [ ] All validation commands pass with zero errors
- [ ] No regressions in existing functionality (234+ existing tests pass)

### Task 5: Implement JWT Token Validator

**File:** `src/identity/token-validator.ts`
**Action:** CREATE
**Description:** JWT validation module using `jose`. Fetches JWKS from provider, verifies JWT signature, validates claims (iss, aud, exp), extracts claims for role mapping. Uses `createRemoteJWKSet` with caching (10 min default). Fail-closed on all validation errors.

**Exports:**

```typescript
export interface TokenValidationResult {
  valid: true;
  claims: JWTPayload;
  subject: string;
}

export interface TokenValidator {
  validate(token: string): Promise<TokenValidationResult>;
}

export function createTokenValidator(config: OAuthConfig): TokenValidator;
```

**Key implementation details:**
- Use `createRemoteJWKSet(new URL(jwksUri))` for JWKS endpoint (auto-discovered from `issuer/.well-known/openid-configuration` if `jwks_uri` not in config)
- Use `jwtVerify(token, jwks, { issuer, audience, clockTolerance })` for validation
- On ANY validation error (expired, bad signature, missing claims): throw `OAuthError`
- Never log the token value — log `subject` and `issuer` only
- Discovery is done once at creation time, JWKS is cached by jose internally

### Task 7: Implement OAuth 2.1 Authorization Code + PKCE Flow

**File:** `src/identity/oauth-flow.ts`
**Action:** CREATE
**Description:** OAuth 2.1 Authorization Code flow with PKCE using `oauth4webapi`. Starts a localhost HTTP callback server, opens the browser for authorization, exchanges the code for tokens, and returns them for storage. Used by `mcp-guard auth login`.

**Exports:**

```typescript
export interface OAuthFlowOptions {
  issuer: string;
  clientId: string;
  clientSecret?: string;
  scopes: string[];
  callbackPort?: number;
}

export interface OAuthFlowResult {
  access_token: string;
  refresh_token?: string;
  id_token?: string;
  expires_at: number;
  scope: string;
}

export async function executeOAuthFlow(options: OAuthFlowOptions): Promise<OAuthFlowResult>;
export async function refreshOAuthToken(
  issuer: string,
  clientId: string,
  refreshToken: string,
  clientSecret?: string,
): Promise<OAuthFlowResult>;
```

**Key implementation details:**
- Use `oauth4webapi.discoveryRequest()` + `processDiscoveryResponse()` for OIDC discovery
- Generate PKCE code verifier with `oauth4webapi.generateRandomCodeVerifier()`
- Start `node:http` server on `127.0.0.1:OAUTH_CALLBACK_PORT` for redirect callback
- Open browser via platform-specific command (`open` on macOS, `xdg-open` on Linux)
- If browser can't open, print URL for manual copy-paste (headless environments)
- Exchange authorization code with `authorizationCodeGrantRequest()`
- Timeout after `OAUTH_CALLBACK_TIMEOUT` (2 min) — throw `OAuthError` on timeout
- Shut down callback server after receiving the code (or on timeout)
- Calculate `expires_at` from `expires_in` response field

  if (!config.url) {

  transport.onerror = (err: Error) => {
    status = 'error';

**Changes:**

```typescript
const authCmd = program
  .command('auth')
  .description('Manage OAuth authentication');

authCmd
  .command('login')
  .description('Authenticate with the configured OAuth provider')
  .option('-c, --config <path>', 'Path to config file')
  .action(async (opts: { config?: string }) => {
    const config = await loadConfig(opts.config);
    if (config.auth.mode !== 'oauth' || !config.auth.oauth) {
      console.error('OAuth is not configured (auth.mode must be "oauth")');
      process.exit(1);
    }
    const result = await executeOAuthFlow({
      issuer: config.auth.oauth.issuer,
      clientId: config.auth.oauth.client_id,
      clientSecret: config.auth.oauth.client_secret,
      scopes: config.auth.oauth.scopes,
    });
    const store = createTokenStore(config.daemon.home);
    await store.save('default', {
      access_token: result.access_token,
      refresh_token: result.refresh_token,
      id_token: result.id_token,
      expires_at: result.expires_at,
      scope: result.scope,
    });
    console.log('Authentication successful — token stored');
  });

authCmd
  .command('status')
  .description('Show current OAuth token status')
  .option('-c, --config <path>', 'Path to config file')
  .action(async (opts: { config?: string }) => {
    const config = await loadConfig(opts.config);
    const store = createTokenStore(config.daemon.home);
    const token = await store.load('default');
    if (!token) {
      console.log('Not authenticated — run `mcp-guard auth login`');
      return;
    }
    const now = Math.floor(Date.now() / 1000);
    const expired = token.expires_at < now;
    const remaining = token.expires_at - now;
    console.log(`Status: ${expired ? 'EXPIRED' : 'Valid'}`);
    if (!expired) {
      console.log(`Expires in: ${Math.floor(remaining / 60)} minutes`);
    }
    console.log(`Scope: ${token.scope}`);
    // Decode JWT header to show subject (without verifying — just for display)
    try {
      const parts = token.access_token.split('.');
      if (parts.length === 3) {
        const payload = JSON.parse(Buffer.from(parts[1], 'base64url').toString());
        console.log(`Subject: ${payload.sub ?? 'unknown'}`);
      }
    } catch { /* opaque token */ }
  });

authCmd
  .command('logout')
  .description('Remove stored OAuth tokens')
  .option('-c, --config <path>', 'Path to config file')
  .action(async (opts: { config?: string }) => {
    const config = await loadConfig(opts.config);
    const store = createTokenStore(config.daemon.home);
    await store.remove('default');
    console.log('Logged out — tokens removed');
  });
```

## Notes

### Design Decisions

1. **OAuth token delivery via `_bearer_token` in params** — mirrors the existing `_api_key` pattern. The bridge injects it, the auth interceptor consumes and strips it. This keeps the bridge thin (no new message types) and the interceptor pipeline uniform.

2. **Both SSE and Streamable HTTP** — The MCP SDK marks `SSEClientTransport` as deprecated in favor of `StreamableHTTPClientTransport`, but many existing MCP servers only speak the older SSE protocol. Supporting both ensures compatibility. The config makes the choice explicit.

3. **OAuth identity overrides OS identity** — In OAuth mode, the daemon still resolves OS identity for the bridge (peer credentials), but the auth interceptor replaces it with the OAuth-derived identity for the pipeline. This is correct: the daemon trusts the bridge process, but the end user is identified by their token.

4. **Token refresh is manual** — `mcp-guard auth login` stores tokens. If they expire, the user runs `auth login` again (or we add auto-refresh in a future PR). The bridge does not refresh — it stays thin.

5. **jose for JWT validation, oauth4webapi for the login flow** — jose handles the hot-path (token validation on every request, with JWKS caching). oauth4webapi handles the cold-path (one-time auth login). Both are lightweight ESM packages.

### Potential Follow-Up Work

- SSE/Streamable HTTP reconnection with configurable backoff (deferred — MCP SDK handles basic reconnection)
- Auto-refresh expired tokens (bridge or daemon-side refresh flow)
- Per-server OAuth config (different providers for different upstream servers)
- Token introspection endpoint support (for opaque tokens, not just JWTs)
- Dashboard showing OAuth session information
- `mcp-guard auth token-info` command for JWT claim inspection
