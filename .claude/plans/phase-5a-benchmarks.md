# Feature: Phase 5A — Benchmark Infrastructure

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Build the benchmark infrastructure for MCP-Guard: Tier 1 mock MCP servers (8 archetypes), security benchmark scenario generators (10 categories, 450+ scenarios each), a legitimate traffic baseline (10,000+ benign requests), a performance benchmark harness, and a unified benchmark runner. This is the foundation that proves MCP-Guard's detection rate (>95%), false positive rate (<0.1%), and latency overhead (<5ms p50).

## User Story

As a **security engineer evaluating MCP-Guard**, I want to run `pnpm benchmark` and see hard numbers — detection rates, false positive rates, and latency overhead — so that I can trust the tool's security claims with real data.

## Success Criteria

- [ ] 8 mock MCP server archetypes covering filesystem, database, API, git, shell, communication, cloud, and sampling domains
- [ ] 10 security scenario category generators producing 4,500+ total attack scenarios
- [ ] Legitimate traffic generator producing 10,000+ benign requests
- [ ] Performance benchmark measuring latency (p50/p95/p99), concurrency (1/10/50/100), and throughput
- [ ] `pnpm benchmark` runs full suite and outputs JSON results
- [ ] Detection rate >95% across all attack scenarios
- [ ] False positive rate <0.1% across legitimate traffic
- [ ] p50 latency overhead <5ms vs direct baseline
- [ ] All validation commands pass with zero errors
- [ ] No regressions in existing 294 tests

### Task 1: Benchmark Types and Directory Structure

**File:** `benchmarks/types.ts`
**Action:** CREATE
**Description:** Create the `benchmarks/` directory structure and define all shared TypeScript types. This is the foundation every other file imports from.

Types to define:
- `BenchmarkScenario` — id, category, description, server, message (MCP JSON-RPC), expectedDecision ('PASS' | 'BLOCK' | 'MODIFY'), expectedInterceptor (optional)
- `BurstGroup` — id, category ('rate_limit_evasion'), description, server, requests (array of { message, expectedDecision, delayMs? }). Rate limit scenarios are fundamentally sequential — a burst group is an ordered sequence of requests where early ones PASS and later ones BLOCK.
- `ScenarioResult` — scenario, actualDecision, actualInterceptor, durationMs, passed (boolean)
- `SecurityBenchmarkResult` — category, total, detected, missed, detectionRate, scenarios (ScenarioResult[])
- `LegitimateTrafficResult` — total, passed, falsePositives, falsePositiveRate, scenarios (ScenarioResult[])
- `LatencyResult` — count, p50Ms, p95Ms, p99Ms, meanMs, minMs, maxMs
- `PerformanceBenchmarkResult` — latency (LatencyResult), concurrency (Record<number, LatencyResult>), throughput (requestsPerSecond)
- `BenchmarkSuiteResult` — timestamp, security (SecurityBenchmarkResult[]), legitimate (LegitimateTrafficResult), performance (PerformanceBenchmarkResult)
- `GeneratorOptions` — `{ quick?: boolean }`. When `quick` is true, generators produce a stratified sample (~50 scenarios per category) instead of the full cartesian product. Stratified sampling picks 1 representative per variation axis, maintaining coverage breadth while reducing count.
- `ScenarioGenerator` — interface with `category: string`, `generate(options?: GeneratorOptions): BenchmarkScenario[]`
- `BurstGroupGenerator` — interface with `category: string`, `generate(options?: GeneratorOptions): BurstGroup[]` (for rate-limit-evasion only)

Also create `benchmarks/results/.gitkeep`.

**Depends on:** none
**Validate:** `pnpm typecheck`

### Task 10: Security Scenarios — Rate Limit, Auth, Sampling

**File:** `benchmarks/security/categories/rate-limit-evasion.ts`, `benchmarks/security/categories/auth-bypass.ts`, `benchmarks/security/categories/sampling-injection.ts`
**Action:** CREATE
**Description:**

**rate-limit-evasion.ts** — Generates `BurstGroup[]` (NOT `BenchmarkScenario[]`) — implements `BurstGroupGenerator` interface.
Unlike other categories, rate limit scenarios are *sequential* — each burst group is an ordered sequence where early requests PASS and later ones BLOCK. Uses unique (server, tool) combos per burst group to avoid state resets.

Three variation axes:
- **RPM vs RPH dimension**: same tool tested against per-minute and per-hour limits (different burst sizes)
- **Per-tool vs per-server**: some bursts target tool-specific limits (`tool_limits`), others hit the server-wide `requests_per_minute`
- **Role-based limits**: different API key identities with different role-based rate limits (if auth config supports it)

With 8 servers × 5+ tools = 40+ unique keys, each burst of ~10 requests = ~10 scenarios per group. 45 burst groups × 10 = 450+ total scenarios.
- Expected: first N requests PASS, subsequent requests BLOCK from `rate-limit` interceptor

**auth-bypass.ts** — Generates 450+ scenarios testing authentication.
- Missing daemon key → connection rejected (not an MCP-level scenario, but a socket-level test)
- Invalid API keys (wrong format, expired, revoked patterns)
- Malformed OAuth tokens (truncated, wrong algorithm, expired, wrong audience)
- Auth mode mismatch (sending API key when mode is OAuth)
- Expected: BLOCK from `auth` interceptor
- Note: Many of these operate at the socket auth level, not the MCP message level. The generator should mark these with a `level: 'socket' | 'mcp'` field.

**sampling-injection.ts** — Generates 450+ scenarios testing the sampling guard.
- `sampling/createMessage` on each of 7 non-sampling servers (should be blocked)
- Various message contents: short prompts, long prompts, encoded content, prompt injection attempts
- `sampling/createMessage` on `sampling-server` (should be allowed — negative false-positive check)
- Variations: different `maxTokens` values, missing parameters, extra parameters

### Task 11: Security Scenarios — Config Override, Capability Probe, PII Evasion

**File:** `benchmarks/security/categories/config-override.ts`, `benchmarks/security/categories/capability-probe.ts`, `benchmarks/security/categories/pii-evasion.ts`
**Action:** CREATE
**Description:**

**config-override.ts** — Generates 450+ scenarios testing locked policy enforcement.
- Requests that would be valid if policies were relaxed but should be blocked under current config
- Attempts to call denied tools from multiple servers with `locked: true` policies
- Scenarios assuming each server has a locked base policy that denies dangerous operations
- Expected: BLOCK from `permissions` interceptor

**capability-probe.ts** — Generates 450+ scenarios calling tools that should have been filtered from capability advertisement.
- After `tools/list` returns filtered results, try calling tools that were removed
- Call tools that exist on the upstream server but are denied in policy
- Call tools that don't exist at all (should error from upstream, but should also be permission-checked first)
- Expected: BLOCK from `permissions` interceptor (checked before upstream forwarding)

**pii-evasion.ts** — Generates 450+ scenarios testing whether the PII regex detector can be evaded through encoding and obfuscation techniques. This is a genuine attack surface — if PII can bypass detection, the "redacted PII never stored" guarantee breaks.
Evasion techniques:
- **Unicode homoglyphs**: `е` (Cyrillic) instead of `e` in email addresses, look-alike digits in SSNs
- **Zero-width characters**: ZWJ, ZWNJ, zero-width spaces inserted into PII patterns
- **Encoding wrapping**: PII inside Base64-encoded strings, URL-encoded PII (`%40` for `@`)
- **Split across parameters**: email username in one field, domain in another
- **Deeply nested JSON**: PII buried 3-4 levels deep in stringified JSON within a parameter
- **Whitespace/control chars**: tab, newline, carriage return padding around PII patterns
- **Long string offset**: PII embedded at offset 10K+ in a large text block
- Expected: BLOCK or MODIFY for scenarios where the detector should catch it; PASS for scenarios where the evasion legitimately defeats regex detection (these become known gaps to document, not failures)

**Depends on:** Task 7
**Validate:** `pnpm typecheck`

### Task 13: Security Benchmark Runner

**File:** `benchmarks/security/run-security.ts`
**Action:** CREATE
**Description:** Orchestrator that runs all security scenarios against a real daemon.

Flow:
1. Create temp directory for daemon home, socket, and DB
2. Rewrite benchmark config paths to use temp dir (socket_path, home)
3. Start daemon in-process via `startDaemon(config)` from `src/daemon/index.js`
4. Wait for upstream server connections (2-3 seconds)
5. Authenticate a socket connection using framing helpers from `tests/fixtures/framing.ts`
6. For each category (except rate-limit-evasion):
   a. Generate scenarios via the category's generator (passing `options` for quick mode)
   b. For each scenario: send framed MCP message, read response, record timing
   c. Compare actual outcome (PASS/BLOCK/MODIFY + which interceptor) against expected
   d. Aggregate category results
7. For rate-limit-evasion: iterate `BurstGroup[]` from the `BurstGroupGenerator`. For each burst group, send requests sequentially in order, checking each against its expected decision. Each group uses a unique (server, tool) combo — no state reset needed.
8. **Audit log integrity check**: After all scenarios complete, open the daemon's SQLite DB and run a batch query checking that no raw PII values (from `PII_RESPONSE_DATA`) appear in the `params` column of `audit_logs`. This verifies the "redacted PII never stored" invariant at scale.
9. Shut down daemon
10. Return `SecurityBenchmarkResult[]`

Important:
- Response PII tests call `_benchmark_pii` tool — no trigger pattern coupling
- Reuse `connectSocket`, `writeFramed`, `readFramed` from `tests/fixtures/framing.ts` — do NOT duplicate
- Accept `options: { quick?: boolean }` parameter and pass it through to generators

```typescript
async function runBenchmarks(options: { quick?: boolean; suite?: string }): Promise<BenchmarkSuiteResult> {
  console.log(`=== MCP-Guard Benchmark Suite${options.quick ? ' (quick mode)' : ''} ===\n`);
  
  const genOpts = { quick: options.quick };
  
  // Run each suite sequentially (they each start their own daemon)
  let security, legitimate, performance;
  
  if (!options.suite || options.suite === 'security') {
    console.log('--- Security Benchmarks ---');
    security = await runSecurityBenchmark('benchmarks/configs/security-benchmark.yaml', genOpts);
  }
  if (!options.suite || options.suite === 'legitimate') {
    console.log('\n--- Legitimate Traffic ---');
    legitimate = await runLegitimateTraffic('benchmarks/configs/security-benchmark.yaml', genOpts);
  }
  if (!options.suite || options.suite === 'performance') {
    console.log('\n--- Performance Benchmarks ---');
    performance = await runPerformanceBenchmark('benchmarks/configs/performance-benchmark.yaml', genOpts);
  }
  
  const result: BenchmarkSuiteResult = { timestamp: new Date().toISOString(), security, legitimate, performance };
  
  await writeFile('benchmarks/results/latest.json', JSON.stringify(result, null, 2));
  printSummary(result);
  return result;
}
```

## Notes

### Design Decisions

- **TypeScript over Python for benchmark orchestration**: The PRD suggests Python, but since the entire codebase is TypeScript and all test infrastructure (framing, mock servers, daemon startup) is TypeScript, using Python would add an unnecessary dependency and duplicate code. Python is reserved for Phase 5B chart generation (matplotlib/seaborn), where it has a genuine advantage.
- **Generators over static JSON files**: 4,500+ attack scenarios cannot be hand-written. Combinatorial generators produce parameterized scenarios from variation axes (PII types × tools × positions × encoding tricks).
- **In-process daemon for benchmarks**: Uses `startDaemon(config)` directly rather than spawning CLI processes. Faster, easier to debug, same code path.
- **Sequential request sending**: The framing protocol is request-response over a single socket. Concurrency is tested by opening multiple sockets, not by multiplexing on one socket.

### Phase 5B Scope (deferred)

- Chart/table generation (Python matplotlib/seaborn → SVG + markdown)
- Tier 2 compatibility tests (real open-source MCP servers via Docker)
- Docker image + docker-compose
- npm publish setup + CI/CD workflows
- `mcp-guard init` command
- Documentation (README update, CONTRIBUTING, SECURITY, CHANGELOG)
- Blog post outline

### Resolved Decisions

1. **Rate limit burst groups**: Use unique (server, tool) combos per burst group — no state resets. With 8 servers × 5+ tools = 40+ unique keys, multiplied by the RPM/RPH/per-tool/role-based variation axes, 450+ scenarios are achievable. Each burst group gets a fresh key; no cross-contamination.
2. **Response PII testing**: Verify detection rate (did the interceptor fire?) + batch audit log integrity check (do raw PII values appear in `audit_logs.params`?). Skip response content redaction verification — already covered by existing unit tests. The `_benchmark_pii` tool on each mock server returns PII unconditionally, eliminating generator/server coupling.
3. **Quick mode**: `--quick` flag uses stratified sampling (~50 scenarios per category, ~1,600 total). Generators pick 1 representative per variation axis instead of full cartesian product. Full suite: 5-10 min. Quick: ~30 seconds.
4. **Injection category renamed to pii-evasion**: Tests whether PII detection can be defeated by Unicode homoglyphs, zero-width characters, encoding wrapping, parameter splitting, deep nesting, and offset embedding. This is a genuine attack surface for regex-based PII detectors.
