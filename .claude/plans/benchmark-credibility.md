# Feature: Benchmark Credibility Overhaul

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

The benchmark suite produces defensible numbers (92–100% detection, 0% FP, 0.21ms p50), but the current presentation lacks the methodological transparency needed to build trust with developers evaluating the tool. This feature overhauls how benchmarks are reported — adding statistical rigor (confidence intervals), detection-scope context (why these numbers are consistent with each other), honest limitations, a coverage gap analysis against academic MCP attack taxonomies, and legitimate traffic diversity metadata — without changing any security logic. The goal is radical transparency: make the methodology so readable that skeptics become contributors.

## User Story

As a security-conscious developer evaluating MCP-Guard, I want the benchmark results to explain *how* and *why* the numbers look the way they do, so that I can trust the methodology rather than just the headline claims.

```typescript
// Add to types.ts:

/** Diversity metadata for legitimate traffic — used in methodology reporting. */
export interface LegitimateTrafficMetadata {
  uniqueServers: number;
  uniqueTools: number;
  nearPiiEdgeCases: number;
  requestCategories: Array<{ name: string; count: number }>;
}

// Add to LegitimateTrafficResult:
//   metadata?: LegitimateTrafficMetadata;
//   fpUpperBound95?: number;  // Rule of Three: 3/n when FP=0

/**
 * Compute the 95% CI upper bound for a proportion.
 * When observed = 0, uses Rule of Three (3/n).
 * When observed > 0, uses Wilson score interval.
 */
export function computeFpUpperBound(observed: number, total: number): number {
  if (total === 0) return 1;
  if (observed === 0) return 3 / total;
  // Wilson score interval upper bound for 95% CI
  const z = 1.96;
  const p = observed / total;
  const denominator = 1 + z * z / total;
  const center = p + z * z / (2 * total);
  const spread = z * Math.sqrt((p * (1 - p) + z * z / (4 * total)) / total);
  return (center + spread) / denominator;
}
```

## Notes

### Design Philosophy: Radical Transparency

The core principle: **make the methodology so transparent that skeptics become contributors.** Open-source security tools earn trust not by having perfect numbers, but by making it trivially easy to verify how those numbers were produced. Every design decision below serves this goal.

### Design Decisions

- **No changes to `src/`**: This is purely a presentation/documentation feature. The interceptor pipeline, PII detection, and audit system are untouched.
- **Rule of Three over Wilson**: For the zero-FP case, the Rule of Three (3/n) is simpler, well-known in biostatistics, and sufficient. Wilson score interval is included for the nonzero case to future-proof.
- **Methodology in REPORT.md is a summary**: The full methodology lives in `docs/benchmark-methodology.md`. REPORT.md includes a condensed version with a link to the full doc.
- **MCPSecBench/MSB are cited, not run against**: These benchmarks test LLM agent resilience (whether the LLM follows malicious instructions), not message-layer proxy detection. Running MCP-Guard against them would require a fundamentally different test harness. Instead, we cite them to show landscape awareness, explain the complementary threat models, and include a coverage gap analysis mapping their attack taxonomies to ours.
- **Self-testing acknowledged, not hidden**: The most damning critique for any security benchmark is "tested against your own test suite." Rather than downplaying this, the methodology doc makes it a prominent section, lists specific mitigations (spot-check tests, structural audit verification, natural variation in results), and explicitly invites community adversarial contributions. Transparency converts a weakness into an invitation.
- **Detection scope explains the numbers as a coherent story**: Sub-ms latency + high detection + zero FP isn't "too good to be true" — it's what you'd expect from deterministic pattern matching and policy enforcement. The methodology ties these together: the latency tells you the detection scope, the detection scope explains the FP rate, and the per-category variation (92-100%) shows the pipeline honestly struggles on harder problems (PII evasion, rate limit gaming). Presenting these as a consistent system, not separate claims, prevents the "lightweight tool" dismissal.
- **Coverage gap analysis against MCPSecBench taxonomy**: Mapping MCPSecBench's 17 attack types to our 10 categories and explicitly showing gaps (network-layer attacks, malicious server detection) demonstrates we know what we don't cover. This is more credible than claiming comprehensive coverage.
- **Leading with per-category breakdown**: The 92-100% range with natural variation is the single most credible element. The "pratfall effect" (minor imperfections increase authenticity) is well-established. This should be more prominent than the headline aggregate.
- **Limitations section in README, not just methodology doc**: Developers evaluating the tool shouldn't have to click through to a separate doc to find honest limitations. Put them right next to the numbers.

### Trade-offs

- **Confidence interval precision vs. simplicity**: Rule of Three gives 3/n ≈ 0.03% for n=10168. A more precise Clopper-Pearson exact interval gives a tighter bound, but the difference is immaterial and Rule of Three is more widely recognized.
- **Methodology doc length**: Target 600-800 words in the standalone doc, 100-150 word summary in REPORT.md. Too short feels like hand-waving, too long feels like protesting too much. The coverage gap analysis table adds length but is high-signal content.
- **Inviting adversarial contributions is a risk**: Someone could submit scenarios that expose genuine gaps. That's the point — it's better to find gaps via community contributions than via production incidents.
