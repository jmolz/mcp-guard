# Feature: Phase 4A — Config Composability + Infrastructure

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Add enterprise config composability (extends with SHA-256 pinning, floor-based merge, locked policies, hot reload), a health HTTP endpoint, SQLCipher encryption at rest, and a dashboard HTTP server shell. This is the first sub-phase of Phase 4 and provides the infrastructure foundation for OAuth (4B) and Dashboard UI (4C).

## User Story

As a **security team lead**, I want to publish a base MCP-Guard config at a URL that developers' personal configs extend via SHA-256-pinned URL, with floor-based merge semantics that prevent them from relaxing my security policies, so that I can enforce consistent security guardrails across the team without managing individual configs.

As an **ops engineer**, I want a health endpoint and encryption at rest, so that I can monitor MCP-Guard in production and meet compliance requirements for data at rest.

## Success Criteria

- [ ] Config `extends` resolves base config from HTTP(S) URL with SHA-256 verification
- [ ] Extends caches base config locally; fetch failure uses cache; no cache + failure → refuse to start
- [ ] Floor-based merge: `allowed_tools` intersected, `denied_tools` unioned, `rate_limit` takes stricter
- [ ] Floor-based merge: PII custom_types additive (can add, cannot remove base)
- [ ] `locked: true` policies cannot be overridden by personal config
- [ ] Hot config reload applies policy changes without daemon restart
- [ ] Hot reload rejects invalid config (keeps previous)
- [ ] Health endpoint returns JSON with daemon status, server statuses, bridge count
- [ ] Dashboard HTTP server starts on configured port with optional token auth
- [ ] SQLCipher encryption derives key from daemon.key, encrypts DB at rest
- [ ] All validation commands pass with zero errors
- [ ] No regressions in existing 179 tests

### Task 3: Create Config Fetcher

**File:** `src/config/fetcher.ts`
**Action:** CREATE
**Description:** HTTP fetch + SHA-256 verification + local caching for `extends` base configs. This module handles the remote config resolution with fail-closed semantics: hash mismatch → ConfigError, fetch failure without cache → ConfigError, fetch failure with cache → use cached copy after re-verifying hash.
**Depends on:** Task 1 (schema types), Task 2 (constants/errors)
**Validate:** `pnpm typecheck && pnpm vitest run tests/config/fetcher.test.ts`

**Exports:**
```typescript
export interface FetchResult {
  yaml: string;
  fromCache: boolean;
}

/** Fetch base config from URL, verify SHA-256, cache locally */
export async function fetchBaseConfig(
  url: string,
  expectedSha256: string,
  cacheDir: string,
): Promise<FetchResult>

/** Compute SHA-256 of content (hex-encoded) */
export function computeSha256(content: string): string
```

**Key implementation details:**
- Use native `fetch()` (Node 22+) with `AbortSignal.timeout(EXTENDS_FETCH_TIMEOUT)`
- SHA-256 via `createHash('sha256').update(yaml, 'utf-8').digest('hex')`
- Cache directory: `{daemon.home}/extends-cache/`
- Cache file named by hash: `{cacheDir}/{sha256}.yaml`
- On fetch success: verify hash → cache → return yaml
- On fetch failure: try cache → verify cached hash → return cached yaml
- On hash mismatch (either live or cached): throw ConfigError (fail-closed)
- Create cache directory with `mkdir(cacheDir, { recursive: true, mode: 0o700 })`

### Task 4: Create Config Merger

**File:** `src/config/merger.ts`
**Action:** CREATE
**Description:** Floor-based config merge — personal configs can only restrict, never relax base policies. This is the security-critical module of Phase 4A. The merge function is structurally enforced, not convention-based.
**Depends on:** Task 1 (schema types)
**Validate:** `pnpm typecheck && pnpm vitest run tests/config/merger.test.ts`

**Exports:**
```typescript
/** Merge personal config onto base config with floor semantics.
 *  Personal config can only restrict, never relax base policies. */
export function mergeConfigs(
  base: McpGuardConfig,
  personal: McpGuardConfig,
): McpGuardConfig
```

**Merge rules (per field):**

| Field Path | Strategy | Implementation |
|-----------|----------|----------------|
| `servers` | Union — personal can add new servers, tighten existing server policies | Deep merge per server; new servers added; existing server policies merged with floor semantics |
| `servers.*.policy.permissions.allowed_tools` | Intersection (personal ∩ base) | Only tools in BOTH lists survive |
| `servers.*.policy.permissions.denied_tools` | Union (personal ∪ base) | All denials from both lists apply |
| `servers.*.policy.permissions.allowed_resources` | Intersection | Same as allowed_tools |
| `servers.*.policy.permissions.denied_resources` | Union | Same as denied_tools |
| `servers.*.policy.rate_limit.requests_per_minute` | Stricter (min of two) | `Math.min(base, personal)` when both defined; base value if only base; personal only when base absent |
| `servers.*.policy.rate_limit.requests_per_hour` | Stricter (min of two) | Same as above |
| `servers.*.policy.sampling.enabled` | AND (both must be true) | `base.enabled && personal.enabled` |
| `servers.*.policy.locked` | Immutable | If `base.policy.locked === true`, personal policy for that server is ignored entirely |
| `pii.actions` | Stricter per type | Personal can escalate (warn→redact, redact→block), cannot relax (block→warn) |
| `pii.custom_types` | Additive | Personal can add types, cannot remove base types |
| `pii.confidence_threshold` | Stricter (lower) | Lower threshold = more sensitive = stricter |
| `auth` | Base wins | Personal cannot change auth mode or configured keys |
| `daemon` | Base wins | Personal cannot change daemon config (port, home, etc.) |
| `interceptors` | Stricter timeout | Lower timeout is stricter |

**Edge cases:**
- Base has `allowed_tools: ['*']`, personal has `allowed_tools: ['read_*']` → result: `['read_*']` (intersection narrows wildcard)
- Base has `denied_tools: ['delete_*']`, personal has `denied_tools: ['drop_*']` → result: `['delete_*', 'drop_*']` (union)
- Base has `policy.locked: true` → entire personal policy for that server is skipped
- PII action severity order: warn (0) < redact (1) < block (2). Personal can only go higher.

   ```typescript
   import { hkdfSync } from 'node:crypto';
   
   export function deriveDbEncryptionKey(daemonKey: Buffer): string {
     const derived = hkdfSync('sha256', daemonKey, '', 'mcp-guard-db-encryption', 32);
     return Buffer.from(derived).toString('hex');
   }
   ```

2. Fix existing SQL injection in encryption pragma:

   ```typescript
   if (options.encryptionKey) {
   ```

## Notes

### Phase 4 Breakdown

Phase 4 (Enterprise Features) is split into sub-phases:

| Sub-Phase | Scope | Dependencies |
|-----------|-------|-------------|
| **4A** (this plan) | Config extends, merge, locked, hot reload, health, SQLCipher | None — builds on Phase 3 |
| **4B** (next) | OAuth 2.1 token auth, role-based permissions, CLI auth commands, SSE transport | Depends on 4A merge semantics |
| **4C** (final) | Dashboard htmx UI, real-time proxy status, request timeline, PII alerts | Depends on 4A health server + 4B identity |

### Dependencies NOT Needed for Phase 4A

- `jose` — only needed for Phase 4B (OAuth JWT validation)
- `oauth4webapi` — only needed for Phase 4B (OAuth introspection)
- `htmx` — only needed for Phase 4C (dashboard views)

Phase 4A uses only existing dependencies + Node.js built-ins.

### Design Decisions

- **HKDF for DB key derivation** (not PBKDF2): The daemon.key is already 256 bits of random, so key stretching isn't needed. HKDF extracts and expands, which is the correct primitive for deriving a separate key from an already-random master secret.
- **Native `fetch()` for extends** (not axios): Node 22+ has stable native fetch. Zero additional dependencies.
- **`fs.watch()` for hot reload** (not chokidar): fs.watch is sufficient for watching a single known file. chokidar adds a large dependency for recursive directory watching we don't need.
- **Health endpoint unauthenticated**: Standard practice for k8s liveness probes. The endpoint exposes only status information, not PII or config details.
- **SQLCipher pragma injection risk**: The existing code uses string interpolation in `db.pragma(\`key='${key}'\`)`. Phase 4A fixes this by validating the key is hex-only before interpolation. SQLCipher's PRAGMA interface doesn't support parameterized queries, so hex validation is the mitigation.

### Risks

1. **Hot reload race condition**: If a request is mid-pipeline when config reloads, the old pipeline reference must complete before being garbage collected. The closure-based reference swap (`currentPipeline = newPipeline`) is safe because JavaScript is single-threaded — the swap happens between event loop ticks, never mid-execution.
2. **Extends circular reference**: If base config A extends B which extends A. Mitigate by: only supporting one level of extends (personal → base), not recursive resolution. The base config cannot itself have an `extends` field.
3. **SQLCipher migration**: Existing unencrypted databases cannot be transparently encrypted. If encryption is enabled on an existing DB, the user must export/import. Phase 4A should detect this and provide a clear error message.
